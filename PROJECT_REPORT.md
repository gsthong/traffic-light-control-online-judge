# Traffic Light Control Online Judge ‚Äî Project Report

## Project Overview
A full-stack web application for evaluating traffic light control algorithms. Users submit Python `control()` functions, and the platform simulates traffic using real physics engines (SUMO/CityFlow) to score their solutions.

---

## Architecture

### Frontend
- **Framework**: React 19 + TypeScript + Vite
- **Styling**: Tailwind CSS v4
- **Animation**: Framer Motion
- **Icons**: Lucide React
- **Rendering**: HTML5 Canvas (520√ó520) with requestAnimationFrame playback loop

### Backend
- **Framework**: FastAPI (Python) + Uvicorn
- **Simulation Engines**:
  - **SUMO** (primary on this machine) ‚Äî real traffic physics via TraCI
  - **CityFlow** (fallback, not installed on Windows)
  - **Pure Python** (fallback ‚Äî custom car-following model)
- **Security**: Subprocess sandbox with import blocking, 2s timeout per controller call

## Key Features Implemented

### 1. Dual-Engine Simulation Backend (`backend/main.py`)
- **CityFlowEngine**: Full CityFlow integration with roadnet.json/flow.json generation
- **SumoEngine**: SUMO via TraCI with netconvert-generated networks, E1 induction loops for sensors
- **Pure Python fallback**: Custom car-following model (IDM-inspired) when no engine is available
- **Strategy pattern**: `get_engine()` factory selects best available engine
- **Coordinate math**: `road_pos_2d()` converts 1D (road, distance) ‚Üí 2D (x, y, angle) with lane offsets

### 2. Secure Sandbox (`run_user_controller`)
- Subprocess isolation with 2s timeout
- Import blocking: `os`, `subprocess`, `socket`, `sys`, `ctypes`, `pickle`, etc.
- UTF-8 encoding support for Windows compatibility
- Temp file cleanup in `finally` block

### 3. React Frontend (`src/App.tsx`)
- **3 tabs**: SIMULATION | SUBMIT | LEADERBOARD
- **Canvas rendering**: Roads, traffic lights, vehicles with proper rotation
- **Coordinate transform**: `screenX = CX + v.x * SCALE`, `screenY = CY - v.y * SCALE` (Y-flip)
- **Playback controls**: Play/Pause, Reset, Speed slider (1x‚Äì20x)
- **Engine selector**: Dropdown to choose Python Sim vs SUMO
- **Auto-playback**: After submit, first scenario auto-selects and plays
- **Leaderboard**: Mock data with user score insertion

### 4. Standalone Evaluator (`backend/evaluator.py`)
- 3600-tick evaluation via stdin/stdout JSON protocol
- **E1 induction loops** for sensor detection (no per-vehicle TraCI loops)
- **Edge-level metrics** via `traci.edge.getWaitingTime()` and `getLastStepHaltingNumber()`
- **Sandbox**: 256MB RAM limit (Linux `resource.setrlimit`), 30s‚Äì1200s wall-clock timeout
- **Error handling**: Full traceback logging, no silent failures

## Technical Challenges Solved

| Problem | Solution |
|---------|----------|
| CityFlow not available on Windows | Dual-engine fallback to SUMO, then pure Python |
| 1500 subprocess spawns per submission (750s+) | Single-subprocess-per-scenario (5 total) |
| SUMO net.xml hand-written ‚Üí invalid | Use `netconvert` from node/edge XML |
| TLS state string length mismatch | Dynamic discovery via `getRedYellowGreenState()` |
| Per-vehicle TraCI calls ‚Üí freeze | E1 induction loops + edge-level metrics |
| Cars floating off canvas | World meters ‚Üí canvas pixel transform with Y-flip |
| `strip("'\"")` syntax error in f-string | `replace("'", "").replace('"', "")` |
| Canvas unmounting on tab switch | Canvas always mounted, only sidebar animates |
| `netgenerate` binary not found | Use `netconvert` with node+edge XML |
| `--tls-files` option doesn't exist | Remove flag, TLS auto-generated by netconvert |
